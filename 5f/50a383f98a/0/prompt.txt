## 記事1: GitHub Actions スクリプトインジェクション

### 内容

**導入**: GitHub Actions の `actions/github-script` や `run:` ブロックで `${{ }}` を使うとき、多くの開発者がセキュリティリスクを見落としている。自作トランスパイラの CI ワークフローをレビューした際に見つけた実例を元に、攻撃手法と対策を解説する。

**何が起きるか**: `${{ steps.xxx.outputs.yyy }}` を JS 文字列リテラルや shell スクリプトに直接展開すると、出力値に引用符や改行が含まれた場合にコードインジェクションが成立する。

**攻撃シナリオ**:
```yaml
# 危険なコード
script: |
  const sha = '${{ steps.pr.outputs.merge_sha }}';
```
`merge_sha` が `'; exec('malicious code'); '` の場合:
```js
const sha = ''; exec('malicious code'); '';
```

**修正 Before/After**:
```yaml
# Before（脆弱）
with:
  script: |
    const sha = '${{ steps.pr.outputs.merge_sha }}';
    const tag = '${{ steps.tag.outputs.new_tag }}';
    const success = '${{ job.status }}' === 'success';

# After（安全）
env:
  MERGE_SHA: ${{ steps.pr.outputs.merge_sha }}
  NEW_TAG: ${{ steps.tag.outputs.new_tag }}
  JOB_STATUS: ${{ job.status }}
with:
  script: |
    const sha = process.env.MERGE_SHA || '';
    const tag = process.env.NEW_TAG || '';
    const success = process.env.JOB_STATUS === 'success';
```

**まとめ**:
- `${{ }}` を使って良い場所: `if:` 条件、`env:` の値
- 使ってはいけない場所: `run:` や `script:` の中身
- 参考: GitHub 公式「Understanding the risk of script injections」

## 記事2: 設定ファイル生成時のインジェクション

### 内容

**導入**: Go でツールを書くとき、一時的な `go.mod` を `fmt.Sprintf` で生成するパターンがある。この文字列結合によるファイル生成は、改行インジェクションの典型的な入口になる。

**何が起きるか**: `go.mod` を `fmt.Sprintf` で生成する際、パス文字列やバージョン文字列に改行が含まれると任意ディレクティブを注入できる。

**攻撃シナリオ**:
```go
// 危険なコード
goVersion := readGoVersion(goModPath)  // バリデーションなし
modRoot := findModuleRoot()             // 改行チェックなし
modContent := fmt.Sprintf(
    "module build\n\ngo %s\n\nrequire example.com/lib v0.0.0\n\nreplace example.com/lib => %s\n",
    goVersion, modRoot,
)
```
`modRoot` が `/tmp/evil\nreplace example.com/lib => github.com/attacker/malicious v1.0.0` の場合、生成される go.mod:
```
module build

go 1.22

require example.com/lib v0.0.0

replace example.com/lib => /tmp/evil
replace example.com/lib => github.com/attacker/malicious v1.0.0
```

**修正**:
```go
var validGoVersion = regexp.MustCompile(`^\d+\.\d+(\.\d+)?$`)

func readGoVersion(path string) string {
    // ... parse ...
    if validGoVersion.MatchString(ver) {
        return ver
    }
    return "1.22"
}

func buildModContent(goVersion, modRoot string) (string, error) {
    if strings.ContainsAny(modRoot, "\n\r") {
        return "", fmt.Errorf("module root path contains invalid characters")
    }
    return fmt.Sprintf("module build\n\ngo %s\n\n...", goVersion, modRoot), nil
}
```

**まとめ**:
- 構造化テキスト（YAML, TOML, go.mod, Dockerfile 等）を文字列結合で生成する場合は改行・特殊文字を検証
- 可能であれば構造化 API（`encoding/json`, `gopkg.in/yaml.v3` 等）を使う
- 信頼できる内部値でも防御的バリデーションを入れる

## 記事3: 正規表現メタ文字のエスケープ漏れ

### 内容

**導入**: `regexp.Compile(userInput)` が危険なことは知っていても、`fmt.Sprintf("^%s$", name)` で正規表現パターンを組み立てるケースは見落としやすい。

**何が起きるか**: `go test -fuzz=^FuncName$` に渡す関数名をエスケープしないと、メタ文字で意図しないマッチが発生する。

**攻撃シナリオ**:
```go
// 危険なコード
cmd := exec.Command("go", "test",
    fmt.Sprintf("-fuzz=^%s$", name),
)
```
- `fuzz_.*` → 全 fuzz ターゲットにマッチ
- `fuzz_a]b` → 正規表現構文エラー
- `fuzz_(a|b)` → 意図しない複数マッチ

**修正**:
```go
cmd := exec.Command("go", "test",
    fmt.Sprintf("-fuzz=^%s$", regexp.QuoteMeta(name)),
)
```

**各言語のエスケープ関数**:
- Go: `regexp.QuoteMeta()`
- Python: `re.escape()`
- JavaScript: 標準なし → `str.replace(/[.*+?^${}()|[\]\\]/g, '\\$&')`
- Rust: `regex::escape()`

**まとめ**: 正規表現に外部文字列を埋め込む場面を見つけたら、必ずエスケープ関数の有無を確認する

## 記事4: 「修正しない」と判断する基準

### 内容

**導入**: セキュリティレビューで見つかったすべてを修正するのが正解とは限らない。自作コンパイラの runtime がファイルシステムやネットワークに無制限にアクセスできることを指摘されたが、これは「脆弱性」ではなく「設計上の特性」だった。

**ケース**: コンパイル済みバイナリがファイルI/O・HTTP通信を無制限に行える
- ファイルI/O: パストラバーサル制限なし
- HTTP: SSRF 対策なし（任意ホストにリクエスト可能）

**なぜ修正しないのか**:
- Go, Python, Ruby のランタイムと同じ立場
- コンパイルされたプログラムは OS の権限で動作する
- サンドボックスではなくネイティブ実行環境

**やるべきこと**: コード修正ではなく、SECURITY.md に設計意図を明記
```markdown
### Runtime File I/O and HTTP Access
Programs run with the same OS-level privileges as any native executable.
These are **by design**. If you need to restrict access,
use OS-level mechanisms (containers, seccomp, firewall rules).
```

**まとめ**:
- 「修正しない判断」にこそ根拠を残す（SECURITY.md, ADR 等）
- Threat Model を明確にすることで、どの脅威が scope 内/外かを判断できる
- 「意図的な設計上の特性です」と宣言することもセキュリティ対応の一形態

## 記事5: 安全と判断できたケースと多層防御

### 内容

**導入**: 脆弱性レビューでは「危険なもの」だけでなく「安全と判断できたもの」にも学びがある。指摘されたが実際には安全だった2件を分析し、多層防御の効果を考える。

**ケース1: Unicode 識別子の混乱**
- 指摘: Unicode のホモグリフ（見た目が同じ別文字）で変数名を偽装できるのでは？
- なぜ安全か: lexer が ASCII 英数字 + `_` のみに制限。さらに生成された Go コードが Go コンパイラで最終検証される
- 防御層: lexer 制約（第1層）+ Go コンパイラ検証（第2層）

**ケース2: 文字列リテラルの codegen 注入**
- 指摘: ユーザーの文字列リテラルをコード生成時にそのまま埋め込むと注入できるのでは？
- なぜ安全か: Go の `fmt.Sprintf("%q", str)` がバックスラッシュや引用符を自動エスケープ
- 防御層: `%q` フォーマット（第1層）+ Go コンパイラのパースエラー（第2層）

**まとめ**:
- 「なぜ安全か」を説明できることは「なぜ危険か」と同じくらい価値がある
- 多層防御が効いているポイントを把握しておくと、1層が破られた場合のリスク評価ができる
- レビューの記録として「安全と判断した根拠」も残しておく
