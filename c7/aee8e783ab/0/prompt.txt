## Code Review: SEO Improvements (seo/improve-search-engine-visibility)

### 1. Sitemap (`website/layouts/_default/sitemap.xml`) -- HIGH

**Issue: Hardcoded 404 path with baseURL prefix is fragile**

Line 9:
```
{{ range where .Site.AllPages "IsHome" false }}{{ if ne .RelPermalink "/meow/404.html" }}
```

The 404 exclusion check uses `"/meow/404.html"` which hardcodes the `baseURL` path prefix (`/meow/`). If the `baseURL` in `hugo.toml` ever changes (e.g., moved to a custom domain where the path is `/` instead of `/meow/`), this filter silently breaks and the 404 page gets included in the sitemap. Hugo's `.RelPermalink` is relative to the `baseURL`, so for this site it will actually be `/meow/404.html`, but this couples the template to the specific deployment path.

**Recommended fix:** Use a more robust approach that does not depend on the base path:
```
{{ range where .Site.AllPages "IsHome" false }}{{ if ne .Kind "404" }}
```
However, Hugo does not expose `Kind` as `"404"` -- the 404 page has `Kind` of `"page"`. A cleaner alternative is to check the `.Layout` or file path:
```
{{ range where .Site.AllPages "IsHome" false }}{{ if not (eq .File.LogicalName "404.html") }}
```
Or more safely, since `.File` may be nil for some page types, use a `RelPermalink` suffix check:
```
{{ if not (strings.HasSuffix .RelPermalink "/404.html") }}
```

**Issue: `.Site.AllPages` includes draft and future pages in development**

The switch from `.Data.Pages` to `.Site.AllPages` is correct for picking up pages from all sections (previously `.Data.Pages` in a sitemap template only returned pages in the current scope). However, `.Site.AllPages` also includes pages of all kinds including taxonomy pages, term pages, and potentially RSS output pages if they exist as page objects. For this small site the practical impact is minimal, but it is worth noting. If taxonomy pages or other page types are added in the future, they will appear in the sitemap without explicit filtering.

**Severity: Medium-High** -- The hardcoded path is the main concern.

### 2. JSON-LD BreadcrumbList (`website/layouts/partials/jsonld.html`) -- MEDIUM

**Issue: BreadcrumbList only handles 2-level depth; no support for 3+ levels**

The current logic handles:
- Home &gt; Section page (depth 2) -- when `.Parent.IsHome` is true
- Home &gt; Section &gt; Leaf page (depth 3) -- when `.Parent` exists and is not home

But if the site ever has content nested 3+ levels deep (e.g., `doc/advanced/topic.md`), the breadcrumb would only show: `Home &gt; doc &gt; advanced/topic` (positions 1, 2, 3), skipping intermediate ancestors. This is acceptable for the current site structure (all content is max 2 levels deep), but worth noting for future-proofing.

**Issue: Duplicated template logic**

Lines 63-70 (the `{{ else }}` branch for when `.Parent` is nil) produce identical output to lines 58-62 (when `.Parent.IsHome` is true). These two branches could be collapsed into one. Having `.Parent` be nil is unusual in Hugo (only the home page has no parent, and the home page is already excluded by the outer `{{ else }}` of `{{ if .IsHome }}`), so this branch may never execute, but the duplication makes the template harder to maintain.

**Issue: `item` on the last BreadcrumbList item**

Per schema.org recommendations, the last item in a BreadcrumbList should ideally omit the `item` property (it represents the current page). Google's structured data documentation says the `item` is optional for the last element. Including it is not an error, but omitting it is considered best practice.

**Severity: Low-Medium** -- Functionally correct for current content, minor improvements possible.

### 3. JSON-LD Article/BlogPosting (`website/layouts/partials/jsonld.html`) -- MEDIUM

**Issue: `image` field may produce an invalid URL when `ogpImage` is empty**

Line 99:
```
"image": "{{ .Site.BaseURL }}{{ .Site.Params.ogpImage }}"
```

If `.Site.Params.ogpImage` is ever unset or empty, this produces `"image": "https://135yshr.github.io/meow/"` which is not a valid image URL. Currently `ogpImage` is set to `"images/ogp.png"` in `hugo.toml`, but there is no guard. Adding a conditional would be safer:

```
{{ with .Site.Params.ogpImage }}"image": "{{ $.Site.BaseURL }}{{ . }}"{{ end }}
```

**Issue: Double trailing slash in publisher logo URL**

Line 96:
```
"url": "{{ .Site.BaseURL }}favicon.svg"
```

Hugo's `.Site.BaseURL` already includes a trailing slash (`https://135yshr.github.io/meow/`), so this produces `https://135yshr.github.io/meow/favicon.svg` which is correct. Good -- no issue here.

**Issue: `Article` type used for section `_index.md` pages**

For section pages (e.g., `/doc/`, `/learn/`), the template emits `"@type": "Article"`. Section index pages are typically navigation hubs, not articles. A more appropriate type might be `"CollectionPage"` or `"WebPage"` for section pages. The `BlogPosting` differentiation (line 77) is correctly applied only to non-section blog pages, which is good. But doc section pages like `/doc/` getting `Article` is semantically questionable.

**Severity: Low-Medium** -- Functional but semantically imprecise.

### 4. Custom 404 Page (`website/layouts/404.html`) -- LOW

**Generally well-done.** A few minor observations:

**Inline styles:** The entire page uses inline `style` attributes rather than CSS classes. This works but is harder to maintain and does not benefit from the site's existing stylesheet. If the design system's CSS variables (e.g., `--color-primary`) change to different names, each inline reference would need individual updating. This is a style/maintainability concern, not a correctness issue.

**Missing `padding: 0` on the `&lt;ul&gt;`:** The `&lt;ul&gt;` element uses `list-style: none` but many browsers apply default `padding-left` to `&lt;ul&gt;` elements. Adding `padding: 0` would prevent unexpected left-spacing in some browsers.

**`{{ define "main" }}` usage is correct:** The 404 template correctly uses `{{ define "main" }}` to slot into `baseof.html`'s `{{ block "main" . }}` block, which means the 404 page gets the full site chrome (header, footer, nav). This is the correct Hugo pattern.

**Severity: Low** -- No functional issues.
